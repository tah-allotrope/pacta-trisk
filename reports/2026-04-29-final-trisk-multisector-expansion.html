01 / Executive Summary

Executive Summary

What changed

The TRISK demo moved from a single hardcoded power flow to a reusable multi-sector pattern that generates inputs, runs sector demos, republishes dashboard snapshots, and exposes a sector selector in the Streamlit UI.

Why it matters

A bank audience can now compare transition stress views for power, cement, and steel without leaving dashboard/pages/2_TRISK_Risk.py, which makes the demo feel like one product instead of three disconnected experiments.

Most important design choice

The implementation kept PACTA as the upstream alignment layer and TRISK as the downstream stress layer, but accepted sector-level SDA context for cement and steel rather than pretending borrower-level alignment evidence existed where the repo does not yet have it.

Recommended direction

Use the current three-sector setup as the production demo baseline, preserve power as the regression reference, and treat any future sector expansion as a repeat of the manifest-plus-sector-spec pattern rather than a return to cloned scripts or flat dashboard files.

02 / Background & Objective

Background & Objective

The starting point was a power-only TRISK pilot. Input preparation in scripts/trisk_prepare_inputs.R, the runner in scripts/trisk_power_demo.R, and the dashboard loader path in dashboard/lib/loaders.py were all structured around one sector and one flat set of CSV outputs.

The objective of the multi-phase effort was to expand that pilot into a broader transition-risk stress demo covering power, cement, and steel while preserving the existing power experience as a regression baseline. The target user outcome was explicit: a bank-facing dashboard where the TRISK page could switch sector views, downloads, tables, charts, and caveats without leaving the same page.

This mattered because the repo already contained the raw ingredients for a broader story: sector rows in data/vietnam_abcd.csv, scenario trajectories in data/vietnam_scenario_co2.csv, company-level power alignment outputs, and sector-level cement and steel SDA outputs. The work was therefore not about inventing a new methodology, but about turning scattered sector evidence into one coherent product surface.

04 / Assumptions & Constraints

Assumptions & Constraints

Assumptions That Enabled Delivery

  • The repo stayed on the newer four-input trisk.model::run_trisk() contract instead of reverting to the older six-CSV interface.
  • data/vietnam_abcd.csv already held enough cement and steel borrower rows to support a first demo covering VICEM, Holcim Group, Hoa Phat Group JSC, and Pomina Group.
  • data/vietnam_scenario_co2.csv served as the v1 scenario source for cement and steel, with emission-intensity pathways translated into TRISK scenario inputs.
  • Power could continue to represent borrower-level market-share alignment, while cement and steel would use sector-level SDA context rather than borrower-specific scores.

Constraints That Shaped The Design

  • The dashboard loader and TRISK page previously assumed one flat file layout, so the data contract and UI contract had to change together.
  • The repo only exposed sector-level cement and steel SDA outputs, which limited how strongly the dashboard could claim borrower-specific combined alignment-and-stress rankings.
  • Power outputs had to remain stable enough to act as a regression baseline during refactoring.
  • The final product needed to stay operator-friendly: reproducible via command-line reruns and discoverable through a manifest instead of filename guesswork.
05 / Methodology

Methodology

The work followed a disciplined sequence from contract definition to full-system validation. First, the plan locked the meaning of each sector through explicit sector_specs, folder conventions, scenario translation rules, and an agreed rule for how cement and steel alignment context would appear next to TRISK outputs. That prevented later code changes from re-deciding sector semantics mid-implementation.

Second, the input builder was refactored so each supported sector could emit the same runnable package shape: assets.csv, financial_features.csv, scenarios.csv, and ngfs_carbon_price.csv. Third, the runner layer was generalized so power remained a stable wrapper while cement and steel could produce the same downstream artifact set.

Fourth, the dashboard snapshot and loader contract were redesigned around dashboard/data/trisk/<sector>/ plus manifest.csv, which moved sector discovery out of page code and into a declared data contract. Finally, the work was closed by replaying the full generation chain, refreshing the snapshot, running pytest smoke coverage, and manually checking Streamlit behavior across all three sectors.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

PHASE-01 / Contract Before Code

The first phase deliberately front-loaded semantics: inventorying cement and steel rows, defining sector mappings, locking the folder structure, and deciding that cement and steel would ship with sector-level SDA context in v1. This phase matters because it turned a fuzzy expansion request into one explicit data and UI contract.

PHASE-02 / Multi-Sector Input Builder

The hardcoded power mapping in scripts/trisk_prepare_inputs.R was replaced with shared sector logic. The result was a reusable builder that could emit inspectable data files and runnable input folders for power, cement, and steel while re-running the power path as a regression check.

PHASE-03 / Shared Runner And Standard Outputs

The power-only runner behavior was extracted into a generic sector runner while scripts/trisk_power_demo.R stayed as the stable compatibility entrypoint. This phase standardized the output contract so each sector could publish the same borrower summaries, result tables, sensitivity outputs, run catalog, and figures.

PHASE-04 / Manifest-Driven Dashboard

The data publishing layer shifted from one flat folder to sector-specific snapshot directories plus manifest.csv. That change enabled dashboard/lib/loaders.py and the TRISK page to discover sectors, switch copy and labels, and export either the selected sector bundle or the full snapshot cleanly.

PHASE-05 / Verification And Handoff

The final phase re-ran the full generation chain, refreshed dashboard data, executed dashboard tests with 9 passed, and manually validated Streamlit sector switching for power, cement, and steel. This phase converted a technically complete refactor into an operationally believable deliverable.

What Changed Across Phases

The project moved from sector-specific code and implicit file naming to a single declared contract. The important transition was not just “more sectors”; it was a shift toward a pattern where each sector becomes a configuration and publishing problem rather than a new script family.

07 / Findings & Recommendation

Findings & Recommendation

Finding 1: The repo now has a coherent three-sector operating model. Inputs, runs, snapshot publication, and dashboard loading all align around one sector-aware contract rather than a collection of power assumptions.

Finding 2: The strongest methodological boundary is still the alignment evidence gap outside power. Power can show borrower-level market-share alignment context, but cement and steel currently rely on sector-level SDA context. The implementation handled that correctly by exposing it as context rather than mislabeling it as borrower-specific truth.

Finding 3: The manifest-based snapshot is the key productization step. It reduced coupling between generated files and the dashboard page, and it creates a repeatable way to add future sectors without teaching UI code about filename conventions.

Rejected alternative: The plan explicitly did not choose per-sector cloned scripts and flat dashboard filenames. That alternative would have delivered short-term speed at the cost of duplicated business logic and fragile UI inference.

Recommendation: Keep the present implementation as the new demo baseline. Treat future sector additions, richer company-level SDA derivations, and any automotive work as follow-on phases built on the current manifest-plus-sector-spec pattern. Do not weaken the labeling discipline that currently distinguishes stress results from sector context.

08 / Implementation Path

Implementation Path

Current Operating Sequence

  1. Run Rscript scripts/trisk_prepare_inputs.R.
  2. Run Rscript scripts/trisk_power_demo.R.
  3. Run Rscript scripts/trisk_sector_demo.R cement.
  4. Run Rscript scripts/trisk_sector_demo.R steel.
  5. Run Rscript scripts/refresh_dashboard_data.R.

Quality Gates

  1. Check that each sector input folder contains the four required TRISK input files.
  2. Check that each synthesis_output/trisk/<sector>_demo/ folder contains the full standardized artifact set.
  3. Run python -m pytest dashboard/tests and keep the smoke suite green.
  4. Run Streamlit and manually switch sectors on the TRISK page before handoff.

Natural Owner Split

Data and scenario maintenance logically sits with the R pipeline surfaces, while presentation and download behavior sit with the dashboard loader and page surfaces. The manifest is the handshake point between those concerns.

Best Follow-On Enhancement

If the next priority is credibility rather than breadth, the highest-value extension is company-level SDA derivation for cement and steel. If the next priority is breadth, the current architecture can accept another sector by extending sector_specs, publishing a new snapshot folder, and updating the manifest.

09 / Risks / Open Questions

Risks / Open Questions

Known Risks

  • Cement and steel results remain more synthetic than power because their pricing and financial assumptions are less mature.
  • Users may over-read repeated SDA context as borrower-level evidence unless labels and caveat text stay explicit.
  • End-to-end reruns can be slow enough that regressions appear late if teams skip sector-by-sector intermediate checks.

Open Questions

  • When should company-level SDA derivation for cement and steel move from deferred research to implemented scope?
  • Should future sector additions keep the same placeholder financial-feature pattern, or is it time to introduce a more explicit assumption registry?
  • When automotive is eventually added, can the current scenario-translation approach generalize cleanly, or will that sector require a new contract layer?
