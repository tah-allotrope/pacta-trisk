01 / Executive Summary

Executive Summary

What was built

A Scenario Builder page (5_Scenario_Builder.py) that exposes the five canonical TRISK parameters as live widgets. Borrowers are ranked by a composite stress-priority score, and the page shows side-by-side baseline vs scenario comparisons, a rank-change delta table, top NPV/PD movers, and save/load/export capabilities.

Two execution modes exist: a precomputed grid (243 scenario combinations × 3 sectors) that works on Streamlit Community Cloud without R, and an operator-only live-rerun path (TRISK_LIVE_RERUN=1) that calls trisk.model::run_trisk() via subprocess for arbitrary parameter combinations.

Key outcomes

  • 23 unit/smoke tests passing on the PHASE-03 page, loader, and scenario-id logic
  • 7 additional live-rerun tests covering env gating, Rscript resolution, and error paths
  • Zero new failures introduced (the single pre-existing smoke-test timeout is unrelated)
  • Demo-ready: A banker can change any lever during a live meeting and immediately see top-10 ranking shifts
  • Exportable: CSV, JSON, and deep-link shareability via ?scenario_id=
02 / Background & Objective

Background & Objective

The PACTA-Vietnam dashboard originally had a read-only TRISK page (2_TRISK_Risk.py) that showed a fixed stress scenario with one-at-a-time sensitivity filtering. A banker watching a live demo could only see pre-canned results and ask “what if” questions that required a developer to rerun the R model.

The objective was to convert that page into a banker-driveable scenario tool so prospects can move shock year, discount rate, risk-free rate, market passthrough, and carbon-price family during a live sales meeting and immediately see how their top-20 borrowers reorder. Two constraints drove the architecture: (1) Streamlit Community Cloud cannot run R, so a public path needed precomputed data; (2) operator-hosted deployments should be able to escape the grid for arbitrary parameter combinations.

03 / Inputs & Scope

Inputs & Scope

In scope

  • Five lever controls: shock_year, discount_rate, risk_free_rate, market_passthrough, carbon_price_family
  • Precomputed grid: 3 values per lever = 243 scenarios × 3 sectors = 729 TRISK runs
  • Side-by-side baseline vs scenario comparison with ranked bars
  • Rank-change delta table and top NPV/PD movers panel
  • Save/load scenarios in session, export as CSV and JSON
  • Deep-link shareability via ?scenario_id= query parameter
  • Operator-only live-rerun adapter gated by TRISK_LIVE_RERUN=1

Out of scope

  • Adding new TRISK sectors (e.g. automotive)
  • Changing TRISK input data or carbon-price curves
  • Multi-bank or multi-portfolio comparison
  • Cross-session persistence of saved scenarios
  • Authentication backend beyond the simple env flag
  • Continuous sliders on the public path (discrete snap-to-grid only)
04 / Assumptions & Constraints

Assumptions & Constraints

Architectural

  • Streamlit Community Cloud cannot run R; all public paths serve from precomputed CSVs and Parquet
  • A factorial grid of 243 combinations × 3 sectors is acceptable (~10 min generation per sector)
  • The dashboard is Python/Streamlit; R is only invoked from the operator-side adapter via subprocess
  • Carbon-price family maps to existing curves; no new NGFS curve files for v1

Operational

  • Public deployment must remain fully synthetic-watermarked; live-rerun mode must never be reachable from the public URL
  • Single-session save/load via st.session_state is adequate for v1
  • 3-value lever discretizations provide meaningful demo comparisons; 5 values would push grid size to 3,125 cells
  • The baseline comparison uses the existing base row from sensitivity_results.csv and the grid’s precomputed delta columns
05 / Methodology

Methodology

The implementation followed the multi-phase plan in plans/2026-05-02-interactive-scenario-builder-plan.md. PHASE-01 and PHASE-02 (contract design and grid generation) were completed before this report. PHASE-03 (the scenario builder page) and PHASE-04 (live-rerun adapter) were implemented in this session.

PHASE-03 built the Streamlit page reusing existing patterns from 2_TRISK_Risk.py and lib/loaders.py. The page loads the precomputed grid from dashboard/data/trisk/grid/<sector>/, converts lever selections to a deterministic scenario_id, and performs an O(1) lookup into the Parquet-backed borrower results table. The baseline view uses the existing sensitivity_results.csv base rows; the scenario view uses the grid slice for the selected scenario_id. Rank changes are computed by diffing independent stress_priority_score rankings.

PHASE-04 wrapped the existing trisk.model::run_trisk() R entrypoint in a thin adapter. scripts/trisk_run_adhoc.R accepts CLI flags, runs one scenario, and writes a temp CSV. dashboard/lib/live_rerun.py subprocesses the script with a 30-second timeout, parses stdout for the output path, loads the CSV, and deletes the temp file. A concurrency guard prevents double-submit. The entire path is invisible when TRISK_LIVE_RERUN=1 is unset.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

PHASE-01 — Contract

Locked the scenario contract: 3 values per lever, scenario_id format s{year}_d{dr}_rf{rf}_mp{mp}_c{family}, and the output file layout for grid/<sector>/{scenarios.csv, borrower_results.parquet, grid_meta.json}.

Key decision: Three NGFS-named aliases mapped to existing repo curves rather than ingesting fresh NGFS Phase V files.

PHASE-02 — Grid Generation

scripts/trisk_scenario_grid.R expanded the contract grid into 243 combinations per sector, ran each via trisk.model::run_trisk(), and wrote Parquet-backed borrower results. Cache-skip behavior keyed on scenario_id for incremental regeneration.

Key decision: 3-value grid (243 cells) accepted as tractable; a 5-value grid would push to 3,125 cells and require parallelization.

PHASE-03 — Scenario Builder Page

New dashboard/pages/5_Scenario_Builder.py with sector selector, five lever widgets (select-sliders + selectbox), side-by-side baseline vs scenario ranked bars, rank-change delta table, top NPV/PD movers panel, save/load via session state, CSV/JSON export, and deep-link via ?scenario_id=. Added load_trisk_grid() to lib/loaders.py.

Edge case caught: _parse_scenario_id initially split on _ which broke for carbon family values containing underscores (e.g. NGFS_NetZero2050). Fixed by using split("_c", 1).

PHASE-04 — Live-Rerun Adapter

scripts/trisk_run_adhoc.R (CLI runner) + dashboard/lib/live_rerun.py (subprocess wrapper) gated behind TRISK_LIVE_RERUN=1. Includes 30-second timeout, Rscript path override via R_RSCRIPT env var, concurrency guard, clear error messages for missing R / missing package, and operator setup docs in docs/streamlit-deploy.md.

Key constraint: The public deployment must never set the env flag; a startup assertion warns when enabled.

07 / Findings & Recommendation

Findings & Recommendation

The two-mode architecture (grid + live rerun) works. The precomputed grid makes the Scenario Builder available on the public Streamlit Cloud URL without any R dependency, which is the highest-leverage marketing surface. The operator-only live-rerun path, while currently behind an env flag, provides a clear upgrade path for private deployments where the sales engineer needs to answer “what if” questions outside the predefined grid.

Tradeoff accepted: The grid is discrete — sliders snap to the 3 preset values. This is a deliberate constraint. Continuous sliders on the public path would require either interpolation (which is misleading for non-linear TRISK outputs) or a full R backend on Streamlit Cloud (which is not possible). The RISK-03-01 mitigation (select-slider with explicit allowed values and a tooltip) was implemented.

Recommendation: Proceed to PHASE-05 for the remaining integration work (demo-script finalization, snapshot refresh documentation, and a full rehearsal walkthrough). The carbon-price family differentiation (all three currently map to the same curve) should be the first post-v1 enhancement, as it’s the lever with the most visible impact on borrower ranking and the one that bankers most often ask about.

09 / Risks / Open Questions

Risks / Open Questions

Risks

  • Accidental live-rerun enablement on public URL: Mitigated by env-var gating, startup assertion, and explicit deployment documentation.
  • Grid generation cost: 3-value grid is currently tractable. Adding a sixth lever would require downsizing existing levers or parallelization.
  • Carbon family homogeneity: All three families currently map to the same curve. Bankers may ask why changing the dropdown doesn’t change results.
  • Pre-existing smoke test timeout: test_trisk_page_renders occasionally times out in CI. Root cause is data-loading latency in Streamlit’s AppTest harness.

Open Questions

  • Should the grid be refreshed automatically on data changes, or is a manual trigger acceptable?
  • Do bankers need a “compare two scenarios” mode (side-by-side of two different lever combinations)?
  • Should the grid be expanded to 5 values per lever (3,125 cells) for finer granularity, or is the 3-value grid adequate for demo purposes?
  • Is the current baseline-comparison method (company_summary base rows vs grid deltas) accurate enough for the intended demo narrative?
